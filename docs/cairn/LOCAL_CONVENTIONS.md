# Cairn Local Conventions

**Audience:** Future CC sessions and human authors of briefs, especially
briefs that prescribe new files, paths, or memory keys inside `D:\claw`.

**Purpose:** Reduce the pattern where briefs guess at paths/names and CC
has to reconcile on arrival. If a convention is documented here, a brief
should either follow it or explicitly call out (and justify) the
departure.

This file is additive — new conventions get appended under the right
heading. Do not rewrite existing entries to match a new preference; if
the convention genuinely changed, log the change with a date.

## Handover notes

**Location:** `docs/cairn/HANDOVER_<YYYY-MM-DD>_<TOPIC>.md`

- Top-level `docs/cairn/`, NOT `projects/<module>/docs/` or
  `projects/cairn/docs/`.
- Date is the session date the handover was written, not the date of the
  work being handed over (usually the same).
- `<TOPIC>` is UPPER_SNAKE_CASE, short, names the work. Examples:
  `AMAZON_LISTINGS`, `CAIRN_DELEGATE`, `BEACON_PHASE_1`.
- One handover per session closeout. Multiple handovers for the same
  session on the same day get disambiguated by adding a suffix
  (`_PART2` etc.), not a different directory.

Examples on disk (2026-04-15):
- `docs/cairn/HANDOVER_2026-04-14_AMAZON_LISTINGS.md`
- `docs/cairn/HANDOVER_2026-04-15_CAIRN_DELEGATE.md`

## Project / codebase naming

**Memory write-backs and retrieval use `project="claw"`.**

- `claw` is the codebase. `D:\claw` on disk. FastAPI on `localhost:8765`.
- Cairn is the ecosystem / brand name, not a project key.
- Do NOT register a `project="cairn"` — the Cairn API reports `cairn`
  as a phantom entry in `/projects` (ignore).
- All of `retrieve_codebase_context`, `retrieve_chat_history`,
  `update_memory` calls for work in this codebase use `project="claw"`.

If a brief prescribes `project="cairn"`, treat it as a mistake and
sed-replace throughout before starting work.

## Code layout (for new features in claw)

Cairn's layout separates concerns cleanly — follow it, don't invent new
top-level directories unless the concern is genuinely new.

- `D:\claw\mcp\cairn_mcp_server.py` — MCP server. Thin JSON-RPC wrapper.
  No business logic. Each tool is a `types.Tool` entry + a dispatch
  branch that POSTs to a FastAPI endpoint with
  `X-API-Key: claw-dev-key-change-in-production`. Add new MCP tools
  here, never in a new server file.
- `D:\claw\api\main.py` — FastAPI route handlers. Check for
  `app.include_router(...)` calls before deciding where a new route
  goes; if routers are split under `api/routers/*.py`, match that
  convention. Otherwise add the route inline in `api/main.py`.
- `D:\claw\core\<domain>\` — business logic. One package per domain
  (`core/models/`, `core/memory/`, `core/social/`, `core/wiki/`,
  `core/delegation/`, etc.). Route handlers in `api/` call into these;
  `core/` never depends on `api/`.
- `D:\claw\data\` — SQLite stores, CSVs, per-project data. Gitignored
  by default; use the `!` patterns in `.gitignore` to whitelist
  templates.
- `D:\claw\scripts\` — one-off scripts, import helpers, Windows
  launchers. Named by what they do, no category prefixes.
- `D:\claw\wiki\modules\` — compiled wiki articles. Tracked in git.
  Regenerated by the wiki compiler; treat as committable output.

**Anti-pattern:** top-level `D:\claw\delegation\` or similar for new
business logic. Business logic goes under `core/`.

## Decision Log

Project-level Decision Log entries live in
`D:\claw\projects\<project>\core.md` under a `## Decision Log` heading.
Entries are ordered by decision id (D-1, D-2, … or D-101, D-102, … for
per-feature decision blocks). Each entry names:

- The decision
- Rationale
- Rejected alternatives (explicit — an empty rejected field is a red
  flag per `CLAUDE.md`)
- Date and session

Large features (e.g., `cairn_delegate`) may additionally have
`wiki/decisions/<feature>.md` for the cross-module decision log — that
is a compiled wiki artefact, not the primary record.

## Wiki articles

Wiki articles live under `D:\claw\wiki\` and are regenerated by the
compiler. Subdirectories:

- `wiki/modules/*.md` — one article per module or module topic.
  Compiled output; tracked in git as deliverable.
- `wiki/patterns/*.md` — cross-cutting patterns (delegation, cost
  discipline, etc.). Hand-authored.
- `wiki/decisions/*.md` — decision records that span modules.
  Hand-authored.
- `wiki/_meta/` — compiler metadata. `last_sync.json` is gitignored.

Compile triggers:
- `POST http://localhost:8765/api/wiki/compile?scope=modules`
- `POST http://localhost:8765/api/wiki/compile?scope=patterns`
- `POST http://localhost:8765/api/wiki/compile?scope=decisions`

## Secrets

- `D:\claw\.env` is the primary secret store for the claw codebase.
  Gitignored. `OPENROUTER_API_KEY` (single word, no underscore between
  OPEN and ROUTER), `ANTHROPIC_API_KEY`, database URLs, etc. all live
  here.
- `.env.example` (tracked) documents the shape without values.
- Never read secret values back into a conversation or write them into
  any committed file. Confirm presence with `grep -c` or env var
  expansion — never echo.

## Commit message convention

Conventional commits, lower-case scope:

- `feat(cairn): ...` — new feature in Cairn itself
- `feat(beacon): ...` — new feature in a module
- `fix(scope): ...` — bug fix
- `docs(scope): ...` — documentation only
- `chore(scope): ...` — housekeeping, dependency bumps, gitignore, etc.
- `refactor(scope): ...` — no behaviour change

One logical change per commit. Trailer `Co-Authored-By:` on
model-assisted commits.
